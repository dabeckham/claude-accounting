The Hidden Tokens You Can Just Ask For

By Don Beckham · 30 June 2026

A reasoning model thinks in tokens you never see — and then throws them away. While chasing down why the model had written one odd little word, I realized those hidden tokens were still sitting in the context, and that I could simply ask for them back. That one request is free to make and surprisingly expensive to keep. Here's the discovery, and what it costs.

TL;DR

  • An extended-thinking model emits a hidden reasoning block before its visible reply. By default that reasoning is ephemeral: billed once as output, used to write the answer, then dropped from the context on the next turn.
  • The aha: while it's still in the current turn's context, you can just ask the model to re-emit it — the hidden tokens print right out. Hidden is not gone; it's one request away.
  • Re-emitting doesn't “double” anything. It changes the reasoning's lifetime: copying it into the visible reply makes it persistent, so it's re-read as cache on every later turn instead of evaporating.
  • The bill is small per block but accretes: every turn's reasoning now rides along forever, climbing the cache_read line and filling the context window faster — so you hit compaction sooner. Worth it to read the model's work; leave it off otherwise.

This one started, as these things do, with a detail too small to matter. The model had used the word iff — logician's shorthand for “if and only if” — in the middle of an otherwise ordinary explanation. I'd never seen a model produce what looked like a typo, and iff looks exactly like a misspelled if. So I asked about it.

1. The word that wasn't a typo

It wasn't a typo; iff is standard notation, used deliberately. But the interesting part was the next question I had: in the entire ocean of training text, there must be a billion ordinary “if”s for every rare “iff.” So how did that rare token win?

The answer is the thing worth keeping. A model doesn't sample from how common a word is overall (its marginal probability). It samples from how likely the word is given everything written above it (its conditional probability). And everything above it was a terse, formal discussion of necessary-and-sufficient conditions — exactly the register where mathematicians reach for iff constantly. Conditioned on that context, iff wasn't a long-shot at all. It was near the top.

A token that's astronomically rare in general can be the obvious, near-deterministic choice locally — once you account for the context it's conditioned on. The gap between those two views is most of what makes a model's output feel uncanny.

2. The aha: the hidden context is right there — ask for it

Here's where it turned. Part of “everything written above” is invisible to me. A reasoning model produces a thinking block — sometimes several — whose text is now hidden in Claude Code. It was a big disappointment when Anthropic started hiding how the model was reasoning, because the visible reply is still conditioned on those hidden tokens — I just never get to see them. I was effectively reconstructing the model's distribution from a partial view of its own prompt.

And then the simple realization landed: if that reasoning is in the context, I can just ask the model to print it. It's not sealed away in some other system — for the duration of the turn it's sitting in the same window everything else is. So I asked it to re-emit the thinking block. It did:

Don makes a subtle point: the conditional context includes the hidden “thinking” portion, which he can't see — which is why the rare token made sense. One honest precision: in this case the visible thread had already locked the formal register, so the hidden part nudged rather than decided…

That's the whole trick. The reasoning I'd been told was hidden was recoverable on demand — not through any special access, just by asking the thing to read its own scratch paper back to me. From there it was a half-step to a standing arrangement: whenever you finish a thinking block, re-emit it before continuing. Suddenly the model's work showed its work.

3. What it costs: ephemeral becomes persistent

The accountant in me couldn't leave it there. If I'm pulling these tokens into every reply, what does that do to the ledger? The intuition is “now everything's in there twice, so it grows faster.” Close, but the real mechanism is sharper — and it's about lifetime, not duplication.

By default, a turn's thinking is ephemeral. It's generated and billed once as output, used to produce the answer, and then — per the documented API behavior — stripped from the context on the next turn. It costs you output tokens exactly once, then zero forever after. It evaporates.

Re-emitting it into the visible reply changes that. The visible message is part of the permanent transcript, and the transcript is re-sent on every following turn (the whole mechanism the companion piece is about). So re-emission doesn't count the tokens twice — it converts tokens that would have vanished into tokens that persist for the rest of the session. The cost lands in two places:

  • Transient, this turn — the hidden block and its visible copy briefly coexist, so that content is generated roughly twice as output. A small, one-time bump.
  • Cumulative, every later turn — the persisted copy is re-read as cache_read (a tenth of the input price) on all subsequent turns, plus a one-time cache write to land it. It never drops back to zero the way the ephemeral original would have.

Put numbers on it. Using the Opus 4.8 list rates — output $25, cache-write-1h $10, cache-read $0.50 per million tokens — for a single 500-token thinking block:

A few cents for one block — trivial in isolation. The catch is the last row's logic applied across the whole session: with re-emission on, every turn's reasoning persists, so the per-turn cache_read base climbs steadily instead of resetting. And as the accounting walk-through shows, cache_read is already the single biggest line on a long session's bill. You're feeding the column that's already the largest.

4. The cost that isn't in dollars

The bigger practical cost isn't the pennies — it's the window. Reasoning that used to be shed each turn now accumulates, so the context fills faster and the session reaches compaction sooner. Compaction costs a summarization pass and gives up fidelity. On a long working session, hitting that wall early usually outweighs the cache-read cents by a wide margin.

Which is the honest punchline for an accounting project: visibility is the product; persistence is the bill. Asking the model to show its work is free to request and genuinely useful — but it quietly trades away the thing that normally keeps long sessions cheap, which is letting the reasoning disappear.

5. So: a switch, not a setting

I keep it off by default and turn it on deliberately — when I actually want to read or audit how a turn reached its answer, or, as here, when the model's thinking is the subject. The toggle is the whole point: pay for the window into the reasoning exactly when the window is worth more than the headroom.

And true to the spirit of this project, the dollar model above is a model, not a measurement. The ledger is the instrument — compare the cache_read line and the context-growth rate across a few turns with re-emission on versus off, and let the numbers settle it. Test, don't theorize.
