Why a Tiny Reply “Costs” 54,000 Tokens

By Don Beckham · 18 June 2026

A walk-through of how per-turn token accounting actually works in an agentic chat session — why a 1,600-character exchange reports tens of thousands of input and cache tokens, how prompt caching makes long conversations cheap (and how to avoid throwing that away), what it costs to serve, and how fast real spend adds up.

TL;DR

  • Every turn re-sends the entire conversation context to the model, so the input and cache token counts reflect the whole session — not your latest message. Only output tracks what you just saw on screen.
  • Prompt caching is what keeps this affordable: a cache read costs $0.50 / M tokens versus $10 / M to write the 1-hour cache — a 20× difference. Claude Code desktop uses the 1-hour cache.
  • That cache refreshes its 1-hour timer every time it's used. If you'll resume the same conversation within the hour, a tiny keep-alive message keeps the whole context at read price; let it lapse and your next message rewrites the entire prefix at 20×.
  • It adds up fast: a single heavy day can top $800 of reference value, the last 7 days came to $2,567, and the trailing 30 days to $3,443 — and the single biggest line item is just cache reads.

This started as a routine glance at my own time- and cost-tracking ledger and turned into a useful tour of how token billing works under the hood. I'm writing it up because the confusion it untangled for me is an extremely common one: the gap between how big a reply looks and how many tokens it reports.

1. The reports that started it

It began with a single command. My very first message of the day was just the accounting skill that I *cough* I mean Claude wrote — /accounting — run as message number one. It ran the report script and printed the report plus a short written summary: about 1,630 characters of text on screen, and nothing else. I read no files into the conversation, pasted no large content, attached no documents. Just a slash-command and its modest output. That's what makes the token figures that follow so surprising at first glance.

That first report, run just after midnight before any real work, showed nothing — as expected:

TOKENS
  in 0  out 0  cache_read 0  cache_write 0

COST  (reference value)
  Actual $0.00    Today $0.00

A second run moments later — after that one short /accounting exchange — looked very different:

SESSIONS
  1 sessions | 1 turns | 2 prompts
  session wall-clock (incl. idle): 1m 58s

TIME
  agent working time (sum of turns): 18s
  your time between turns (idle/read): 1m 40s

COST  (reference value)
  Actual $0.28    Today $0.28

TOKENS
  in 13,702  out 578  cache_read 54,090  cache_write 16,524

2. The key idea: only one number tracks your reply

The intuition “small reply → small token count” is reasonable, but it doesn't hold for a stateless chat API. Here is the crucial split:

out is the only number tied to the visible reply. Everything else is the entire conversation context being re-processed on that turn.

  • out 578 — this is the reply. About 1,630 characters ÷ 578 ≈ 2.8 characters per token. That checks out almost exactly.
  • in 13,702, cache_read 54,090, cache_write 16,524 — these are the input side: the whole context sent to the model to produce those 578 output tokens.

Every API call re-sends the full context: the system prompt, all tool schemas, the assistant's operating instructions, the available-skills list, project configuration and memory, and the entire prior conversation. That is tens of thousands of tokens before the user types a single character. In an agentic session with a large toolset, a 50,000-token stable prefix on every turn is ordinary.

What the cache fields mean

FieldValueWhat it actually is
out578The generated reply text (~1,630 chars ÷ 2.8 ≈ 578 ✓)
in13,702Uncached input tokens sent this turn
cache_read54,090Stable prefix served from cache — the cheap path, a tenth of input price
cache_write16,524New content written into the cache this turn

The large cache_write is the tell-tale sign of an early turn: the cache was cold, so a big chunk of context was written to it for the first time (a cache miss becomes a cache write). On later turns that prefix is warm, so cache_read climbs and cache_write shrinks. A third report later in the session showed exactly that:

TOKENS
  in 14,111  out 7,686  cache_read 354,346  cache_write 26,671

cache_read jumped to 354K (the now-warm prefix, read back each turn) while cache_write stayed flat — and out climbed to 7,686, because that's the one figure that tracks the visible replies, and the replies had gotten longer.

The ledger code bears this out: the token figures aren't computed from the reply text at all. They are summed straight from the usage block the API itself records for each assistant message — input_tokens, output_tokens, cache_read_input_tokens, and cache_creation_input_tokens — over the messages since the last user prompt. Those are the provider's own billed counts, not an estimate.

3. The cache is the whole game — don't let it expire

If every turn re-sends a 50K–700K-token prefix, why isn't every turn ruinously expensive? Prompt caching. The first time the model sees a stable block of context it writes it to a cache; on later turns that same block is read back at a fraction of the price instead of being re-processed from scratch.

Here is the pricing that makes the strategy obvious. These are the list rates for the model in use here, Claude Opus 4.8, per million tokens:

Token typeRate / 1Mvs. inputNotes
Input (uncached)$5.001×Fresh tokens the model must read in full
Output$25.005×Generated tokens (the reply)
Cache write — 5-min TTL$6.251.25×Short-lived cache
Cache write — 1-hour TTL$10.002×What Claude Code desktop uses
Cache read (hit)$0.500.1×Re-using already-cached context

Two facts do all the work here:

  • A cache read is 20× cheaper than a 1-hour cache write ($0.50 vs $10.00 per million), and 10× cheaper than sending the same tokens as fresh uncached input.
  • Claude Code desktop writes to the 1-hour cache, and a cache's timer is refreshed every time it is used. (Verified directly from the ledger: across a recent week, every cache-write token landed in the 1-hour bucket and none in the 5-minute bucket.)

The practical rule: if you'll continue the same conversation within the hour, keep the cache warm. A trivial keep-alive message — even just “.” — reads the existing context (cheap) and resets the 1-hour timer. Let the hour lapse and your next message has to rewrite the entire prefix at 20× the read price — so the break-even is about 19 hours away (one rewrite ≈ 20 hourly keep-alive reads, and the prefix size cancels out).

What that's worth, concretely. Take a long-running session with a 200,000-token cached prefix — ordinary for sustained agentic work:

  • Keep it warm — one keep-alive read of the prefix: 200,000 × $0.50/M = $0.10, and the clock resets for another hour.
  • Let it expire — the next message is a cache miss and must rewrite the whole prefix: 200,000 × $10/M = $2.00.

So a single lapse-and-rewrite on a 200K context costs about $2.00, versus roughly $0.10 to have kept it alive — and bigger contexts scale that gap proportionally. Pinging once an hour to bridge a gap stays cheaper than one cold rewrite until you're idle for the better part of a day; past that, just let it go. The rule of thumb: don't let an active conversation's cache expire unless you're genuinely done with it for many hours — it is cheaper to tap the cache and reset the timer than to pay to rebuild it an hour or two later.

(One caveat: this only helps if you intend to keep using the same conversation. If you're finished with it, let it expire — a brand-new session starts with a cold cache regardless.)

4. So how fast does this add up?

Caching keeps the per-message cost low, but sustained agentic work runs many turns across many sessions and several projects, and the totals climb fast. These are reference-value figures straight from the ledger (under a subscription login), across a mix of models:

PeriodReference-value spendNotes
A quiet day (24 May)$1.89A few short sessions — almost nothing
A heavy day (16 Jun)$827.05The single biggest day on record
Last 7 days$2,567.0435 sessions, 351 turns
Last 30 days$3,443.37History back to 22 May

One heavy day broke $800 of reference value on its own. And notice where the money actually goes. Narrowing to just the Opus 4.8 turns over the last 7 days (so a single set of rates applies) and decomposing by token type:

Token typeVolumeRate / 1MCost
Input (uncached)1,591,045$5.00$7.96
Output13,119,997$25.00$328.00
Cache write (1-hour)73,148,872$10.00$731.49
Cache read2,502,322,553$0.50$1,251.16
Total$2,318.61

The headline: cache reads alone are $1,251 — more than half the Opus bill for the week — even though, per token, they're the cheapest thing on the menu. There were simply 2.5 billion of them. That is the same fact as section 3 seen from the billing side: the context is read back over and over, and the only reason 2.5 billion token-reads cost $1,251 instead of ~$25,000 is that they were cache hits at $0.50/M, not rewrites at $10/M. Uncached input, by contrast, is a rounding error ($7.96). Keep the cache warm and this is the column you're protecting.

5. So what does it actually cost to serve?

Heavy caveat first: these are order-of-magnitude estimates with wide error bars. Providers don't publish serving costs, and the real figure moves with batch size, hardware utilization, GPU generation, and electricity contracts. Treat the numbers as “which power of ten,” not precision.

The idea that matters: not all tokens cost the same to serve. Decode (output) is the expensive part — it is sequential and memory-bandwidth bound. Prefill (input), cache writes, and especially cache reads are far cheaper; a cache read is essentially a memory fetch with almost no compute. A rough, all-in serving cost — including amortized GPU hardware, not just electricity — applied to one of the lighter days ($0.71 of reference value):

CategoryTokensEst. cost rateEst. cost
Output7,686~$2–10 / M~$0.02–0.08
Input (prefill)14,111~$0.1–0.5 / M~$0.00
Cache write26,671~$0.1–0.5 / M~$0.01
Cache read354,346~$0.02–0.1 / M~$0.01–0.04
Total~$0.05–0.15

So serving a $0.71 reference-value day plausibly costs the provider somewhere around 5–15 cents in compute, power, and hardware amortization — on the order of 10–20% of the API-equivalent price, implying a healthy gross margin on API list rates.

Where the floor comes from: an 8-GPU serving node draws roughly 10 kW. With batching it might emit on the order of 1,000–2,000 output tokens per second across all concurrent requests. That works out to only about $0.10–0.30 of electricity per million output tokens. Serving cost lands higher than that purely because of GPU capital amortization and real-world utilization well below 100% — not because of the electrons.

Then why the “losing money on subscriptions” talk?

That debate is not cost-versus-retail — as the table shows, API list prices sit comfortably above serving cost. It's flat-fee versus retail-value:

  • A subscription is a flat monthly fee. A $20/month plan is roughly $0.67/day.
  • A heavy user can easily generate hundreds of dollars of reference value per day — the $827 single-day figure above is on a subscription login, not an API bill.
  • At that point the subscriber consumes far more retail-equivalent value than the flat fee covers. The provider may still be above serving cost — it costs cents on the dollar — but below the API price they'd otherwise charge. So it's a loss against API revenue, not necessarily a cash loss on compute.

That's exactly why this ledger labels cost as “reference value” under a subscription login: it measures what the tokens would cost at API rates — the right yardstick for the subscription-economics question — not the provider's actual serving cost and not the user's flat fee.

The limit of that nuance: my own usage

That last clause — not necessarily a cash loss — holds for ordinary heavy use. My own usage crossed it on both plans I've held. I started on the $20 tier and upgraded to $100 (its rate limits run about 5× the $20 tier's) at mid-afternoon on 12 June; splitting the ledger at that exact moment:

PeriodPlanActive daysReference valueEst. COGS to serve
22 May – 12 Jun$20/mo13$1,035$39–193
12 Jun – now$100/mo7$2,422$86–430

Even on the $20 plan I was deep underwater: ~$1,035 of reference value and an estimated $39–193 of real compute over three weeks — against roughly $14 of fees for that span. Upgrading didn't fix the arithmetic; it raised the ceiling. The 5× limits let me more than quadruple my daily burn ($80 → $346 of reference value per active day), and a single week ran $86–430 of compute against ~$23 of prorated fee. At both tiers the serving cost alone runs several times — up to ~20× — what I actually pay. The higher plan didn't make me profitable to serve; it just enlarged the loss.

And it isn't only Anthropic. Microsoft is in the same boat with GitHub Copilot — GPT-5-class tokens cost on the same order as Opus tokens, so the flat-fee math doesn't close on either side. The strain shows: as of this writing, every paid Copilot plan greets new users with “We're gradually enabling new sign-ups. If you're unable to sign up right now, please check back soon” — rationing access rather than selling unlimited use. Flat-rate plans inevitably grow usage caps, premium-request quotas, or sign-up throttles, because the per-token economics underneath don't close.
